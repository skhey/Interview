# Node.js Clustering & Load Balancing – Complete Guide

## Why Clustering Matters

Node.js runs on a **single-threaded event loop**. By default, a Node.js process can only utilize **one CPU core**. On modern multi-core servers (8, 16, 32+ cores), this means 90%+ of available CPU capacity sits idle.

**Clustering** solves this by spawning multiple Node.js processes (workers) that share the same server port. Each worker runs on its own event loop, allowing the application to handle more concurrent requests and fully utilize all CPU cores.

```
┌─────────────────────────────────────────────────────────────┐
│                     Master Process (PID 1000)               │
│  - Manages workers                                          │
│  - Does NOT handle requests                                 │
│  - Restarts dead workers                                    │
└──────────────────────┬──────────────────────────────────────┘
                       │
         ┌─────────────┼─────────────┬─────────────┐
         ▼             ▼             ▼             ▼
    ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
    │ Worker 1│  │ Worker 2│  │ Worker 3│  │ Worker 4│
    │ CPU 0   │  │ CPU 1   │  │ CPU 2   │  │ CPU 3   │
    │ Event   │  │ Event   │  │ Event   │  │ Event   │
    │ Loop    │  │ Loop    │  │ Loop    │  │ Loop    │
    └─────────┘  └─────────┘  └─────────┘  └─────────┘
         │             │             │             │
         └─────────────┴─────────────┴─────────────┘
                       │
              Shared Port (e.g., 3000)
```

**When to use clustering:**
- CPU-bound workloads (image processing, encryption, compression)
- High-traffic web servers needing maximum throughput
- Applications running on multi-core servers

**When NOT to use clustering:**
- Single-core containers (e.g., t2.micro on AWS)
- I/O-bound apps where the event loop is mostly idle
- When already using container orchestration (Kubernetes) that scales pods horizontally

## The Built-in `cluster` Module

Node.js provides a native `cluster` module for process management.

### Basic Cluster Setup

```javascript
const cluster = require('node:cluster');
const http = require('node:http');
const os = require('node:os');

const numCPUs = os.cpus().length;  // Number of CPU cores

if (cluster.isMaster) {
  console.log(`Master ${process.pid} is running`);
  
  // Fork workers — one per CPU core
  for (let i = 0; i < numCPUs; i++) {
    cluster.fork();
  }
  
  // Handle worker exits
  cluster.on('exit', (worker, code, signal) => {
    console.log(`Worker ${worker.process.pid} died. Restarting...`);
    cluster.fork();  // Auto-restart on crash
  });
  
} else {
  // Worker process — handles HTTP requests
  http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Handled by worker ${process.pid}\n`);
  }).listen(3000);
  
  console.log(`Worker ${process.pid} started`);
}
```

**Key observations:**
- All workers call `.listen(3000)` on the **same port**
- The master process **does not listen**; it delegates incoming connections to workers
- Each worker runs in its own isolated V8 instance with separate memory

### How Port Sharing Actually Works

When multiple processes call `listen()` on the same port, the operating system would normally throw `EADDRINUSE`. The `cluster` module bypasses this:

1. The **master process** creates the actual server socket and binds to the port
2. When a worker calls `listen()`, it receives a **file descriptor reference** to the master's socket (via IPC)
3. The OS kernel distributes incoming connections among all processes listening on that socket

```
    Client Request ──▶ Kernel (TCP Socket :3000) ──▶ Round-Robin Distribution
                                                              │
                                    ┌─────────────────────────┼─────────────────────────┐
                                    ▼                         ▼                         ▼
                               Worker 1                  Worker 2                  Worker 3
```

**This distribution is handled by the OS kernel, not Node.js** (except on Windows, which uses a custom round-robin scheduler).

## Forking Multiple Worker Processes

### Dynamic Worker Management

```javascript
const cluster = require('node:cluster');
const os = require('node:os');

class ClusterManager {
  constructor(options = {}) {
    this.workers = new Map();
    this.maxWorkers = options.maxWorkers || os.cpus().length;
    this.restartDelay = options.restartDelay || 1000;
    this.workerRestarts = new Map(); // Track restart count per worker ID
  }
  
  start() {
    if (!cluster.isMaster) return;
    
    console.log(`Master ${process.pid} starting ${this.maxWorkers} workers`);
    
    // Fork initial workers
    for (let i = 0; i < this.maxWorkers; i++) {
      this.forkWorker();
    }
    
    // Listen for worker messages
    cluster.on('message', (worker, message) => {
      this.handleWorkerMessage(worker, message);
    });
    
    // Handle worker exits
    cluster.on('exit', (worker, code, signal) => {
      this.handleWorkerExit(worker, code, signal);
    });
    
    // Graceful shutdown on master signals
    process.on('SIGTERM', () => this.shutdown());
    process.on('SIGINT', () => this.shutdown());
  }
  
  forkWorker() {
    const worker = cluster.fork({
      WORKER_ID: `worker-${Date.now()}-${Math.random().toString(36)}`
    });
    
    this.workers.set(worker.id, {
      instance: worker,
      startTime: Date.now(),
      requestsHandled: 0
    });
    
    worker.on('message', (msg) => {
      if (msg.type === 'request-count') {
        const info = this.workers.get(worker.id);
        if (info) info.requestsHandled = msg.count;
      }
    });
    
    console.log(`Forked worker ${worker.process.pid} (ID: ${worker.id})`);
    return worker;
  }
  
  handleWorkerExit(worker, code, signal) {
    const workerInfo = this.workers.get(worker.id);
    const uptime = workerInfo ? Date.now() - workerInfo.startTime : 0;
    
    console.error(`Worker ${worker.process.pid} died. Code: ${code}, Signal: ${signal}, Uptime: ${uptime}ms`);
    
    this.workers.delete(worker.id);
    
    // Check if we should restart (avoid infinite restart loops)
    const restartCount = this.workerRestarts.get(worker.id) || 0;
    
    if (restartCount > 5) {
      console.error(`Worker ${worker.id} crashed >5 times. Not restarting.`);
      this.workerRestarts.delete(worker.id);
      return;
    }
    
    this.workerRestarts.set(worker.id, restartCount + 1);
    
    // Delay restart to avoid CPU thrashing
    setTimeout(() => {
      if (this.workers.size < this.maxWorkers) {
        this.forkWorker();
      }
    }, this.restartDelay);
  }
  
  handleWorkerMessage(worker, message) {
    switch (message.type) {
      case 'ready':
        console.log(`Worker ${worker.process.pid} ready`);
        break;
      case 'error':
        console.error(`Worker ${worker.process.pid} error:`, message.error);
        break;
      case 'health':
        // Store health metrics
        break;
    }
  }
  
  shutdown() {
    console.log('Master shutting down...');
    
    // Notify all workers to gracefully exit
    for (const [id, info] of this.workers) {
      info.instance.send({ type: 'shutdown' });
    }
    
    // Wait for workers to exit (max 10 seconds)
    let remaining = this.workers.size;
    const timeout = setTimeout(() => {
      console.error('Workers did not exit gracefully. Force killing...');
      for (const [id, info] of this.workers) {
        info.instance.kill('SIGKILL');
      }
      process.exit(1);
    }, 10000);
    
    cluster.on('exit', () => {
      remaining--;
      if (remaining === 0) {
        clearTimeout(timeout);
        console.log('All workers exited. Master exiting.');
        process.exit(0);
      }
    });
  }
  
  getMetrics() {
    const metrics = [];
    for (const [id, info] of this.workers) {
      metrics.push({
        workerId: id,
        pid: info.instance.process.pid,
        uptime: Date.now() - info.startTime,
        requests: info.requestsHandled
      });
    }
    return metrics;
  }
}

// Usage
const manager = new ClusterManager({ maxWorkers: os.cpus().length });
manager.start();
```

**Worker code (same file, executed in worker context):**


## Sticky Sessions (Session Affinity)

### The Problem with Stateless Load Balancing

By default, the OS distributes connections **round-robin** among workers. This works perfectly for stateless applications. But if your app maintains **in-memory session state** (e.g., user logged in, shopping cart), the same user may hit different workers on subsequent requests, losing their session.

```
Request 1 ──▶ Worker 1 (sets session in memory)
Request 2 ──▶ Worker 2 (doesn't have session) ❌ User logged out!
```

### Solution: Sticky Sessions

Sticky sessions ensure that **all requests from a particular client go to the same worker**.

#### Implementation with `sticky-session` library

```javascript
const cluster = require('node:cluster');
const http = require('node:http');
const sticky = require('sticky-session');

const server = http.createServer((req, res) => {
  // Session handling
  const sessionId = req.headers.cookie?.match(/sessionId=([^;]+)/)?.[1];
  
  res.writeHead(200);
  res.end(`Worker ${process.pid} | Session: ${sessionId || 'none'}`);
});

if (!sticky.listen(server, 3000)) {
  // Master process
  server.once('listening', () => {
    console.log('Master started');
  });
} else {
  // Worker process
  console.log(`Worker ${process.pid} started`);
}
```

**How `sticky-session` works:**
1. The master process receives all connections
2. It hashes the client's IP address (or a custom header/cookie)
3. It forwards the connection to the same worker for that hash
4. The worker handles the actual HTTP request

#### Manual Sticky Session Implementation

```javascript
const cluster = require('node:cluster');
const net = require('node:net');
const crypto = require('node:crypto');

if (cluster.isMaster) {
  const workers = [];
  const numWorkers = require('node:os').cpus().length;
  
  // Fork workers
  for (let i = 0; i < numWorkers; i++) {
    const worker = cluster.fork();
    workers.push(worker);
  }
  
  // Master creates the TCP server
  const server = net.createServer({ pauseOnConnect: true }, (socket) => {
    // Get client IP for hashing
    const clientIP = socket.remoteAddress;
    
    // Hash IP to consistently pick a worker
    const hash = crypto.createHash('md5').update(clientIP).digest('hex');
    const workerIndex = parseInt(hash.substring(0, 8), 16) % workers.length;
    const selectedWorker = workers[workerIndex];
    
    // Send socket to selected worker
    selectedWorker.send('sticky-session:connection', socket);
  });
  
  server.listen(3000, () => {
    console.log(`Master listening on port 3000`);
  });
  
} else {
  // Worker process
  const http = require('node:http');
  
  // Worker creates HTTP server
  const server = http.createServer((req, res) => {
    res.writeHead(200);
    res.end(`Worker ${process.pid}\n`);
  });
  
  // Receive socket from master
  process.on('message', (msg, socket) => {
    if (msg === 'sticky-session:connection') {
      server.emit('connection', socket);
      socket.resume();
    }
  });
}
```

### Sticky Sessions with Redis (Distributed Session Store)

A better approach for production is to **avoid sticky sessions entirely** by storing session data externally:

```javascript
const session = require('express-session');
const RedisStore = require('connect-redis')(session);
const redis = require('redis');

const redisClient = redis.createClient({
  url: process.env.REDIS_URL
});

app.use(session({
  store: new RedisStore({ client: redisClient }),
  secret: process.env.SESSION_SECRET,
  resave: false,
  saveUninitialized: false,
  cookie: { secure: process.env.NODE_ENV === 'production' }
}));
```

**Benefits:**
- Any worker can handle any request
- Sessions survive worker crashes/restarts
- Scales horizontally across multiple servers

## PM2 Cluster Mode

[PM2](https://pm2.keymetrics.io/) is a production process manager that simplifies clustering and adds monitoring, log management, and zero-downtime reloads.

### Starting in Cluster Mode

```bash
# Start app with clustering (auto-detects CPU count)
pm2 start app.js -i max

# Start with specific number of instances
pm2 start app.js -i 4

# Start with a name
pm2 start app.js --name "my-api" -i max
```

### PM2 Ecosystem File (`ecosystem.config.js`)

```javascript
module.exports = {
  apps: [{
    name: 'production-api',
    script: './server.js',
    instances: 'max',           // Use all CPU cores
    exec_mode: 'cluster',        // Enable cluster mode
    watch: false,               // Disable in production
    max_memory_restart: '1G',   // Restart if memory exceeds 1GB
    env: {
      NODE_ENV: 'production',
      PORT: 3000
    },
    error_file: '/var/log/api/error.log',
    out_file: '/var/log/api/out.log',
    log_date_format: 'YYYY-MM-DD HH:mm:ss Z',
    merge_logs: true,
    // Graceful shutdown settings
    kill_timeout: 10000,        // Wait 10s for connections to close
    listen_timeout: 5000,       // Wait 5s for app to listen
    // Auto-restart on crash
    autorestart: true,
    max_restarts: 10,
    min_uptime: '10s',
    // Load balancing
    instance_var: 'INSTANCE_ID'
  }]
};
```

### Zero-Downtime Reload with PM2

```bash
# Gracefully reload all processes (zero downtime)
pm2 reload all

# Or reload specific app
pm2 reload production-api
```

**How `pm2 reload` works:**
1. Spawns new worker processes with the updated code
2. New workers start and begin accepting connections
3. Old workers stop accepting new connections
4. Old workers finish processing existing requests
5. Old workers are gracefully terminated

### PM2 Monitoring Commands

```bash
# List all processes
pm2 list

# Monitor CPU/Memory in real-time
pm2 monit

# Show logs
pm2 logs production-api
pm2 logs production-api --lines 100

# Show detailed info
pm2 show production-api

# Save current process list (auto-restart on server reboot)
pm2 save
pm2 startup
```

### Sticky Sessions with PM2

**Option 1: Use external session store (recommended)**

**Option 2: Use PM2 in fork mode with Nginx as reverse proxy**

```nginx
        proxy_set_header Connection 'upgrade';
```

## Inter-Process Communication (IPC) in Clusters

Workers can communicate with the master and with each other via message passing.

### Worker → Master

```javascript
// In worker
process.send({ type: 'metric', value: requestCount });

// In master
cluster.on('message', (worker, message) => {
  if (message.type === 'metric') {
    console.log(`Worker ${worker.id} handled ${message.value} requests`);
  }
});
```

### Master → Worker

```javascript
// In master
worker.send({ type: 'reload-config' });

// In worker
process.on('message', (msg) => {
  if (msg.type === 'reload-config') {
    loadNewConfiguration();
  }
});
```

### Worker-to-Worker (via Master)

```javascript
// Worker A wants to broadcast
process.send({ type: 'broadcast', data: 'Cache invalidated' });

// Master relays
cluster.on('message', (sender, message) => {
  if (message.type === 'broadcast') {
    for (const id in cluster.workers) {
      const worker = cluster.workers[id];
      if (worker.id !== sender.id) {
        worker.send(message);
      }
    }
  }
});
```

## Real-World Production Example

Complete production-ready Express app with clustering:

```javascript
// server.js
const cluster = require('node:cluster');
const os = require('node:os');
const express = require('express');
const helmet = require('helmet');
const compression = require('compression');
const Redis = require('ioredis');

const numCPUs = process.env.NODE_ENV === 'production' 
  ? os.cpus().length 
  : 1;

if (cluster.isMaster) {
  console.log(`Master ${process.pid} starting ${numCPUs} workers`);
  
  const workers = new Map();
  
  for (let i = 0; i < numCPUs; i++) {
    const worker = cluster.fork();
    workers.set(worker.id, {
      pid: worker.process.pid,
      startTime: Date.now(),
      requests: 0
    });
  }
  
  cluster.on('exit', (worker, code, signal) => {
    console.error(`Worker ${worker.process.pid} died. Restarting...`);
    
    // Remove from tracking
    workers.delete(worker.id);
    
    // Fork new worker
    const newWorker = cluster.fork();
    workers.set(newWorker.id, {
      pid: newWorker.process.pid,
      startTime: Date.now(),
      requests: 0
    });
  });
  
  cluster.on('message', (worker, message) => {
    if (message.type === 'request') {
      const info = workers.get(worker.id);
      if (info) info.requests++;
    }
  });
  
  // Metrics endpoint for monitoring
  const metricsServer = express();
  metricsServer.get('/metrics', (req, res) => {
    const metrics = [];
    for (const [id, info] of workers) {
      metrics.push({
        workerId: id,
        pid: info.pid,
        uptime: Date.now() - info.startTime,
        requests: info.requests
      });
    }
    res.json({
      master: process.pid,
      workers: metrics,
      totalWorkers: workers.size
    });
  });
  
  metricsServer.listen(9090, () => {
    console.log('Metrics server on port 9090');
  });
  
  // Graceful shutdown
  const shutdown = () => {
    console.log('Master shutting down...');
    
    for (const id in cluster.workers) {
      cluster.workers[id].send({ type: 'shutdown' });
    }
    
    let remaining = Object.keys(cluster.workers).length;
    const timeout = setTimeout(() => {
      console.error('Force killing workers...');
      process.exit(1);
    }, 30000);
    
    cluster.on('exit', () => {
      remaining--;
      if (remaining === 0) {
        clearTimeout(timeout);
        console.log('All workers exited.');
        process.exit(0);
      }
    });
  };
  
  process.on('SIGTERM', shutdown);
  process.on('SIGINT', shutdown);
  
} else {
  // Worker process
  const app = express();
  
  // Redis client for sessions/cache
  const redis = new Redis(process.env.REDIS_URL);
  
  // Security & performance middleware
  app.use(helmet());
  app.use(compression());
  app.use(express.json());
  
  // Session configuration (using Redis)
  const session = require('express-session');
  const RedisStore = require('connect-redis')(session);
  
  app.use(session({
    store: new RedisStore({ client: redis }),
    secret: process.env.SESSION_SECRET,
    resave: false,
    saveUninitialized: false,
    cookie: {
      secure: process.env.NODE_ENV === 'production',
      httpOnly: true,
      maxAge: 24 * 60 * 60 * 1000 // 24 hours
    }
  }));
  
  // Request counter middleware
  app.use((req, res, next) => {
    process.send({ type: 'request' });
    next();
  });
  
  // Health check endpoint
  app.get('/health', (req, res) => {
    res.json({
      status: 'healthy',
      worker: process.pid,
      uptime: process.uptime(),
      memory: process.memoryUsage()
    });
  });
  
  // Main routes
  app.get('/', (req, res) => {
    res.json({
      message: 'Hello from worker',
      pid: process.pid,
      sessionId: req.sessionID
    });
  });
  
  // Graceful shutdown handler
  process.on('message', (msg) => {
    if (msg.type === 'shutdown') {
      console.log(`Worker ${process.pid} shutting down...`);
      
      server.close(async () => {
        await redis.quit();
        process.exit(0);
      });
      
      setTimeout(() => process.exit(1), 10000);
    }
  });
  
  const server = app.listen(3000, () => {
    console.log(`Worker ${process.pid} listening on port 3000`);
  });
}
```

## Summary Table

| Feature | Built-in `cluster` | PM2 Cluster Mode |
|---------|-------------------|------------------|
| **Setup Complexity** | Manual code required | One-line command |
| **Auto-Restart on Crash** | Manual implementation | Built-in |
| **Zero-Downtime Reload** | Complex to implement | `pm2 reload` |
| **Log Management** | Manual | Automatic with rotation |
| **Monitoring Dashboard** | None | `pm2 monit`, Keymetrics |
| **Memory Limit Auto-Restart** | Manual | `max_memory_restart` |
| **Sticky Sessions** | Manual via custom code | Use Nginx in front |
| **Production Readiness** | Requires additional tooling | Production-ready out-of-box |

**Best Practices:**
