# Node.js Internal Architecture: Libuv & V8 Engine — Complete Guide

Node.js sits at the intersection of two powerful systems: **libuv**, which handles all asynchronous I/O and the event loop, and the **V8 engine**, which compiles and executes JavaScript. Understanding how these internals work together is essential for debugging performance issues, optimizing throughput, and building production‑grade Node.js applications.

---

## Part 1: libuv — The Asynchronous I/O Engine

libuv is a **cross‑platform C library** originally developed for Node.js to abstract away operating system differences in asynchronous I/O. It provides:

- A **full‑featured event loop** backed by the best I/O polling mechanism available on each platform (`epoll` on Linux, `kqueue` on macOS/BSD, `IOCP` on Windows)
- A **thread pool** for offloading blocking operations that cannot be performed asynchronously at the OS level
- Cross‑platform abstractions for file system operations, networking (TCP/UDP sockets), DNS resolution, and child processes

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Node.js Application                                │
│                    (JavaScript code running on main thread)                   │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                                  libuv                                       │
│  ┌─────────────────────────────────────────────────────────────────────────┐│
│  │                           Event Loop (Single Thread)                     ││
│  │   timers → pending → idle/prepare → poll → check → close callbacks      ││
│  └─────────────────────────────────────────────────────────────────────────┘│
│                                      │                                        │
│                    ┌─────────────────┼─────────────────┐                      │
│                    ▼                 ▼                 ▼                      │
│            ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
│            │ epoll/kqueue │  │  Thread Pool │  │    DNS       │              │
│            │    /IOCP     │  │ (default: 4) │  │   (getaddrinfo)│              │
│            └──────────────┘  └──────────────┘  └──────────────┘              │
│                    │                 │                 │                      │
│                    ▼                 ▼                 ▼                      │
│            ┌──────────────┐  ┌──────────────┐  ┌──────────────┐              │
```

### The Event Loop: Phases in Depth

The libuv event loop processes callbacks in a specific order through six distinct phases. Each phase has its own FIFO queue of callbacks.

```
   ┌───────────────────────────┐
┌─>│           timers          │  setTimeout(), setInterval() callbacks
│  └─────────────┬─────────────┘
│                ▼
│  ┌───────────────────────────┐
│  │     pending callbacks     │  I/O callbacks deferred from previous cycle
│  └─────────────┬─────────────┘
│                ▼
│  ┌───────────────────────────┐
│  │      idle, prepare        │  Internal libuv housekeeping (rarely used directly)
│  └─────────────┬─────────────┘
│                ▼
│  ┌───────────────────────────┐
│  │           poll            │  Retrieve new I/O events; execute I/O callbacks
│  └─────────────┬─────────────┘
│                ▼
│  ┌───────────────────────────┐
│  │          check            │  setImmediate() callbacks
│  └─────────────┬─────────────┘
│                ▼
│  ┌───────────────────────────┐
│  │      close callbacks      │  socket.on('close', ...), cleanup handlers
│  └─────────────┬─────────────┘
│                │
└────────────────┘
```

**1. Timers Phase**
Executes callbacks scheduled by `setTimeout()` and `setInterval()`. The timer's threshold is the **minimum** time after which the callback will run — not a guarantee. If the event loop is blocked, timers will be delayed.

**2. Pending Callbacks Phase**
Executes I/O callbacks that were **deferred** to the next loop iteration. For example, TCP errors or certain system operation callbacks are queued here.

**3. Idle / Prepare Phase**
Internal libuv phases used for housekeeping. Not exposed to JavaScript.

**4. Poll Phase — The Heart of the Loop**
The poll phase has two main responsibilities:
- **Calculate how long to block** and wait for I/O (based on the nearest timer)
- **Process events in the poll queue**: execute callbacks for incoming connections, data on sockets, file read completions, etc.

If the poll queue is empty and there are timers scheduled, the loop will wait until the next timer expires. If there are `setImmediate()` callbacks scheduled, the poll phase will **stop and proceed to the check phase** immediately.

**5. Check Phase**
Executes `setImmediate()` callbacks. `setImmediate()` is specifically designed to run **immediately after the poll phase**, before the loop continues.

**6. Close Callbacks Phase**
Executes callbacks for closed resources: `socket.on('close')`, `fs.close()`, etc.

### Microtasks: The Hidden Queue

Before moving from one phase to the next, the event loop **empties the microtask queue**:
- `process.nextTick()` callbacks (highest priority)
- `Promise` resolutions (`.then`, `.catch`, `.finally`)

```
┌───── Phase ─────┐     ┌───── Phase ─────┐     ┌───── Phase ─────┐
│                  │────▶│                  │────▶│                  │
│  Process queue   │     │  Process queue   │     │  Process queue   │
│        │         │     │        │         │     │        │         │
│        ▼         │     │        ▼         │     │        ▼         │
│  Microtasks ✅   │     │  Microtasks ✅   │     │  Microtasks ✅   │
│  (nextTick then  │     │  (nextTick then  │     │  (nextTick then  │
│   Promises)      │     │   Promises)      │     │   Promises)      │
└──────────────────┘     └──────────────────┘     └──────────────────┘
```

**Critical implication:** Recursive `process.nextTick()` calls can **starve the event loop** because they run before proceeding to the next phase.

### The Thread Pool: Offloading Blocking Work

Not all I/O can be performed asynchronously at the OS level. **File system operations**, **DNS lookups**, and **crypto/compression** functions use **blocking system calls**. libuv offloads these to a **thread pool** to prevent them from stalling the main event loop.

| Operation | Handled By | Blocking? |
|-----------|------------|-----------|
| TCP/UDP sockets | epoll/kqueue/IOCP | No (non‑blocking) |
| File I/O (`fs.readFile`) | Thread Pool | Blocks a thread |
| DNS (`dns.lookup`) | Thread Pool | Blocks a thread |
| Crypto (`crypto.pbkdf2`) | Thread Pool | Blocks a thread |
| Zlib compression | Thread Pool | Blocks a thread |
| `dns.resolve*()` | libuv's own thread | No (uses c‑ares) |

**Default Thread Pool Size: 4**

The default size is **4 threads**, which means only **4 blocking operations** can run in parallel before queuing occurs. This becomes a bottleneck for I/O‑heavy applications.

**Tuning the Thread Pool:**
```bash
# Increase thread pool size (e.g., to 8)
UV_THREADPOOL_SIZE=8 node server.js
```

```javascript
// You can also set it programmatically (before any async operations)
process.env.UV_THREADPOOL_SIZE = 8;
```

**Real‑World Impact:**
```javascript
const crypto = require('node:crypto');
const start = Date.now();

// Launch 6 pbkdf2 operations (CPU‑intensive hashing)
for (let i = 0; i < 6; i++) {
  crypto.pbkdf2('password', 'salt', 100000, 64, 'sha512', () => {
    console.log(`Hash ${i + 1}: ${Date.now() - start}ms`);
  });
}

// Output with UV_THREADPOOL_SIZE=4:
// Four hashes complete at ~500ms, the remaining two at ~1000ms
// Because only 4 threads process at a time

// Output with UV_THREADPOOL_SIZE=6:
// All six complete at ~500ms
```

### How File I/O Works (Step‑by‑Step)

```javascript
const fs = require('node:fs');

fs.readFile('/path/to/large-file.txt', (err, data) => {
  console.log('File read complete');
});
console.log('Reading file...');
```

**What happens under the hood:**
1. `fs.readFile` is called. V8 passes the request to Node.js core.
2. Node.js delegates to libuv.
3. libuv sees this is a file operation (not a socket) and **assigns it to the thread pool**.
4. A worker thread calls the blocking OS `read()` syscall.
5. The main event loop **continues running** (logs "Reading file...").
6. When the worker thread finishes, it places the callback in the **pending callbacks queue**.
7. The event loop picks up the callback in the next `pending callbacks` phase and executes it.

### How Socket Operations Work (epoll)

```javascript
const http = require('node:http');

const server = http.createServer((req, res) => {
  res.end('Hello World');
});

server.listen(3000);
```

**What happens under the hood:**
1. `server.listen()` creates a non‑blocking TCP socket.
2. libuv registers this socket with the OS's I/O notification system:
   - **Linux:** `epoll` — monitors thousands of file descriptors efficiently
   - **macOS/BSD:** `kqueue`
   - **Windows:** `IOCP` (I/O Completion Ports)
3. When a client connects, the OS notifies libuv via the registered mechanism.
4. libuv places the connection event in the **poll queue**.
5. The event loop processes it during the poll phase.

### Scheduling: `setTimeout` vs `setImmediate` vs `nextTick`

```javascript
// Understanding the execution order
const fs = require('node:fs');

fs.readFile(__filename, () => {
  setTimeout(() => console.log('timeout'), 0);
  setImmediate(() => console.log('immediate'));
  process.nextTick(() => console.log('nextTick'));
  Promise.resolve().then(() => console.log('promise'));
});

// Output order:
// nextTick
// promise
// immediate    (or timeout — depends on system)
// timeout      (or immediate)

// Inside an I/O callback, setImmediate ALWAYS runs before setTimeout
```

**Why?**
- `process.nextTick` and Promises are microtasks — they run **immediately after the current operation**, before the next event loop phase.
- Inside an I/O callback (poll phase), `setImmediate` is scheduled for the **next check phase**, while `setTimeout` is scheduled for the **next timers phase**.
- The check phase occurs **immediately after** the poll phase, so `setImmediate` fires before the next loop iteration's timers.

### Real‑World: Detecting Event Loop Lag

```javascript
const { monitorEventLoopDelay } = require('node:perf_hooks');

const histogram = monitorEventLoopDelay({ resolution: 20 });
histogram.enable();

setInterval(() => {
  const stats = histogram;
  console.log({
    min: stats.min / 1e6,
    max: stats.max / 1e6,
    mean: stats.mean / 1e6,
    p99: stats.percentile(99) / 1e6,
  });
  histogram.reset();
}, 5000);

// This shows you when the event loop is being blocked
// Useful for detecting CPU‑intensive work on the main thread
```

## Part 2: V8 Engine — JavaScript Execution & Optimization

V8 is Google's high‑performance JavaScript engine, written in C++. It compiles JavaScript directly to native machine code and manages memory automatically via garbage collection.

### The V8 Compilation Pipeline

V8 uses a **multi‑tier compilation strategy** to balance startup time, memory usage, and peak performance.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           JavaScript Source Code                             │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                                 Parser                                      │
│                     (Generates Abstract Syntax Tree - AST)                   │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │
                                      ▼
┌─────────────────────────────────────────────────────────────────────────────┐
│                         Ignition (Interpreter)                               │
│               Converts AST to bytecode, starts execution immediately         │
│               Collects type feedback (profiling data) during execution       │
└─────────────────────────────────────┬───────────────────────────────────────┘
                                      │
                    ┌─────────────────┼─────────────────┐
                    │                 │                 │
                    ▼                 ▼                 ▼
         ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐
         │    Sparkplug     │ │    Maglev        │ │    TurboFan      │
         │  (Non‑optimizing │ │  (Mid‑tier JIT)  │ │  (Optimizing JIT)│
         │    compiler)     │ │  (Node.js 22+)   │ │                  │
         └──────────────────┘ └──────────────────┘ └──────────────────┘
                    │                 │                 │
                    └─────────────────┼─────────────────┘
                                      ▼
                          ┌────────────────────────┐
                          │    Native Machine Code │
                          └────────────────────────┘
```

**1. Ignition (Interpreter)**
When JavaScript is first executed, Ignition converts the AST to **bytecode** and starts running it immediately. This provides **fast startup**. While interpreting, Ignition collects **type feedback** (profiling data) about how the code is actually being used — which types appear, which branches are taken.

**2. Sparkplug (Non‑Optimizing Compiler)**
Introduced to bridge the gap between Ignition and TurboFan. Sparkplug compiles bytecode to machine code **very quickly** without heavy optimization, providing a speed boost for code that hasn't become "hot" enough for TurboFan.

**3. Maglev (Mid‑Tier JIT)**
Added in Node.js 22, Maglev sits between Sparkplug and TurboFan, generating optimized machine code faster than TurboFan but with less aggressive optimizations.

**4. TurboFan (Optimizing Compiler)**
When a function is identified as **hot** (called many times), TurboFan uses the collected type feedback to generate **highly optimized machine code**. This is where V8 achieves near‑native performance.

**Key Insight:** If type feedback becomes invalid (e.g., a function that always received numbers suddenly receives a string), V8 **deoptimizes** — it throws away the optimized machine code and falls back to the interpreter. This is expensive.

### Hidden Classes (Shapes / Maps)

JavaScript objects are dynamic — properties can be added, removed, or changed at runtime. V8 cannot know an object's "shape" ahead of time. To make property access fast, V8 creates internal blueprints called **Hidden Classes** (also called **Shapes** or **Maps**).

```
// When you write:
const obj1 = { x: 1, y: 2 };
const obj2 = { x: 3, y: 4 };

// V8 creates:
// ┌─────────────────────┐     ┌─────────────────────┐
// │   Hidden Class C0   │────▶│   Hidden Class C1   │
// │   (empty object)    │     │   { x: offset 12 }  │
// └─────────────────────┘     └──────────┬──────────┘
//                                        │ add property 'y'
//                                        ▼
//                              ┌─────────────────────┐
//                              │   Hidden Class C2   │
//                              │ { x: offset 12,     │
//                              │   y: offset 20 }    │
//                              └─────────────────────┘
//
// obj1 and obj2 share C2 — property access is a direct memory offset
```

Each hidden class records:
- The **names** of properties
- Their **memory offsets** within the object

When two objects have the same properties added in the same order, they **share the same hidden class**. Property access then becomes a simple **pointer arithmetic** operation (base + offset) rather than a dictionary lookup.

**The Hidden Class Transition Chain:**

```javascript
const obj = {};           // Hidden class C0 (empty)
obj.x = 10;               // Transition to C1 (has 'x')
obj.y = 20;               // Transition to C2 (has 'x', 'y')
delete obj.x;             // ⚠️ DANGER: C2 → C3 (deprecated)

// V8 cannot reuse C1 because the transition history is broken
// This makes the object slower
```

**Optimization Rules for Hidden Classes:**
1. **Initialize all properties in the constructor** — avoid adding properties after creation
2. **Always add properties in the same order** — `{ a: 1, b: 2 }` and `{ b: 3, a: 4 }` get different hidden classes
3. **Avoid `delete`** — it breaks the transition chain and creates a slow "dictionary mode" object
4. **Keep property types consistent** — don't change a number property to a string

```javascript
// ❌ Bad: Different hidden classes
class Point {
  constructor(x, y) {
    this.x = x;
    // y added later in some cases
  }
  setY(y) {
    this.y = y;  // Objects without y have different shape
  }
}

// ✅ Good: All properties initialized in constructor
class Point {
  constructor(x, y) {
    this.x = x;
    this.y = y || 0;  // All Point objects have same shape
  }
}
```

### Inline Caching (IC)

Even with hidden classes, V8 still needs to **verify** that the object's hidden class matches the expected one each time a property is accessed. **Inline Caching** caches this verification step.

```
// Function that accesses a property
function getX(obj) {
  return obj.x;
}

// First call: V8 records that obj had hidden class C1, offset=12
getX({ x: 1 });

// Second call with same hidden class: V8 uses cached offset directly
getX({ x: 2 });  // Fast!

// Third call with different hidden class: Cache miss, V8 updates IC
getX({ x: 3, y: 4 });  // Still works, but slower

// Many different shapes → "megamorphic" → IC stops optimizing
```

**IC States:**
| State | Description | Performance |
|-------|-------------|-------------|
| **Uninitialized** | First time seeing this property access | Slow (needs to discover) |
| **Monomorphic** | Always sees the same hidden class | **Fastest** (direct offset) |
| **Polymorphic** | Sees 2‑4 different hidden classes | Moderate |
| **Megamorphic** | Sees >4 different hidden classes | **Slowest** (dictionary lookup) |

**Optimization for Monomorphic IC:**
```javascript
// ❌ Bad: Many different shapes passed to same function
function process(obj) {
  return obj.value;
}
process({ value: 1 });
process({ value: 2, extra: true });
process({ value: 3, another: 'x' });
// IC becomes polymorphic/megamorphic

// ✅ Good: Consistent object shapes
class Data {
  constructor(value) {
    this.value = value;
  }
}
const items = [new Data(1), new Data(2), new Data(3)];
items.map(process);  // Stays monomorphic
```

### V8 Memory Management: Garbage Collection

V8 manages memory in a **managed heap** divided into generations.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                              V8 Managed Heap                                │
│  ┌─────────────────────────────────────────────────────────────────────────┐│
│  │                         New Space (Young Generation)                     ││
│  │   Size: 1‑8 MB | Uses: Scavenger (Cheney's algorithm — copy survivors)  ││
│  │   Collects: Frequently, very fast                                       ││
│  └─────────────────────────────────────────────────────────────────────────┘│
│                                      │                                        │
│                    Survives two minor GCs → promoted                          │
│                                      ▼                                        │
│  ┌─────────────────────────────────────────────────────────────────────────┐│
│  │                         Old Space (Old Generation)                       ││
│  │   Size: Up to heap limit | Uses: Mark‑Sweep & Mark‑Compact              ││
│  │   Collects: Less frequently, can be slower                              ││
│  └─────────────────────────────────────────────────────────────────────────┘│
│                                                                              │
│  ┌──────────────────────┐  ┌──────────────────────┐  ┌─────────────────────┐ │
│  │      Large Object    │  │     Code Space       │  │    Map Space        │ │
│  │        Space         │  │  (compiled code)     │  │  (hidden classes)   │ │
│  └──────────────────────┘  └──────────────────────┘  └─────────────────────┘ │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Minor GC (Scavenge)**
- Runs frequently on **New Space** (young generation)
- Uses **copying collection**: live objects are copied to a survivor space, dead objects are simply abandoned
- Extremely fast because it only touches live objects
- Objects that survive **two minor GCs** are **promoted** to Old Space

**Major GC (Mark‑Sweep & Mark‑Compact)**
- Runs on **Old Space** when it fills up
- **Mark phase**: Traces all reachable objects from roots (global object, stack, closures)
- **Sweep phase**: Frees memory of unmarked objects
- **Compact phase**: Defragments memory by moving objects together (prevents fragmentation)

**Incremental & Concurrent Marking**
To avoid long pauses (stop‑the‑world), V8 uses **incremental marking**: the GC marks objects in small chunks interleaved with JavaScript execution. Recent versions also use **concurrent marking** (runs in background threads).

**Orinoco — V8's Modern GC**
V8's garbage collector is codenamed **Orinoco**. It coordinates:
- Concurrent marking (background threads)
- Incremental marking (interleaved)
- Parallel scavenging (multiple threads during minor GC)
- Concurrent sweeping (background threads)

### Memory Tuning Flags

```bash
# Set heap limits
node --max-old-space-size=4096 server.js   # 4 GB old space
node --max-semi-space-size=64 server.js    # 64 MB new space

# GC logging (for debugging)
node --trace-gc server.js
node --trace-gc-verbose server.js

# Expose GC for manual triggering (development only!)
node --expose-gc server.js
# Then call global.gc() in code
```

### Real‑World: V8 Performance Optimization Checklist

| Issue | Symptom | Fix |
|-------|---------|-----|
| **Megamorphic ICs** | Functions slow down over time | Ensure consistent object shapes |
| **Hidden class transitions** | Slow property access | Initialize all props in constructor |
| **`delete` operator** | Object goes to dictionary mode | Set to `null`/`undefined` instead |
| **`arguments` object** | Prevents optimization | Use rest parameters `...args` |
| **`try...catch` in hot functions** | Blocks TurboFan optimization | Move error handling outside hot path |
| **Large heap, frequent GC** | High CPU, latency spikes | Increase `--max-old-space-size` |

## How libuv and V8 Work Together

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Node.js Request Lifecycle                         │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. Incoming HTTP request arrives at socket                                 │
│                              │                                               │
│                              ▼                                               │
│  2. libuv epoll detects event, places callback in poll queue                │
│                              │                                               │
│                              ▼                                               │
│  3. Event loop picks up callback, passes to V8                              │
│                              │                                               │
│                              ▼                                               │
│  4. V8 executes JavaScript handler (Ignition → maybe TurboFan)              │
│                              │                                               │
│                              ▼                                               │
│  5. Handler calls fs.readFile() → V8 calls Node.js core → libuv             │
│                              │                                               │
│                              ▼                                               │
│  6. libuv sends file read to thread pool. Event loop continues.             │
│                              │                                               │
│                              ▼                                               │
│  7. Thread completes, callback queued in pending callbacks                  │
│                              │                                               │
│                              ▼                                               │
│  8. Event loop picks up callback, V8 executes it                            │
│                              │                                               │
│                              ▼                                               │
│  9. Response sent via socket (libuv writes, OS sends)                       │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Key Insight:** libuv handles **everything asynchronous** (I/O, timers, sockets). V8 handles **everything synchronous** (executing JavaScript, computing, memory management). The main thread alternates between:
1. Running V8 (JavaScript execution)
2. Running libuv (event loop tick, polling for I/O)

If V8 spends too long in JavaScript (blocking), libuv's event loop is **starved** — timers are delayed, I/O callbacks queue up, and the server becomes unresponsive.

## Summary Table

| Concept | libuv | V8 |
|---------|-------|-----|
| **Primary Role** | Asynchronous I/O and event loop | JavaScript execution engine |
| **Thread Model** | Single‑threaded event loop + thread pool | Single‑threaded execution + background GC threads |
| **Key Components** | Event loop, thread pool, epoll/kqueue/IOCP | Ignition, Sparkplug, Maglev, TurboFan, Orinoco GC |
| **Performance Critical** | Thread pool size, event loop blocking | Hidden classes, inline caches, GC pauses |
| **Tuning Flags** | `UV_THREADPOOL_SIZE` | `--max-old-space-size`, `--max-semi-space-size` |

**Key Takeaways:**
1. **libuv abstracts platform differences** — same async code runs on Linux, macOS, Windows
2. **Thread pool is for blocking operations** — file I/O, crypto, DNS, compression
3. **Default thread pool size is 4** — increase for I/O‑heavy apps
4. **Hidden classes optimize property access** — keep object shapes consistent
5. **Inline caching makes repeated property access fast** — stay monomorphic
6. **V8 uses generational GC** — short‑lived objects are cheap, long‑lived objects are expensive
7. **Event loop phases determine execution order** — `setImmediate` vs `setTimeout` depends on context
8. **Microtasks run between phases** — recursive `nextTick` can starve the loop
