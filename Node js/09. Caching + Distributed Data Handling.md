# Caching & Distributed Data Handling – Complete Guide

Caching is one of the most effective ways to improve application performance, reduce database load, and handle high traffic. In distributed systems, caching strategies become critical for maintaining data consistency while scaling horizontally. This guide covers everything from simple in‑memory caches to complex distributed caching architectures.

---

## 1. In‑Memory Caching

In‑memory caching stores frequently accessed data in the application's RAM for sub‑millisecond access times. It's the **fastest cache tier** but limited by process memory and lack of distribution.

### Simple Object Cache (Not Recommended for Production)

```javascript
// ❌ Basic but problematic: no TTL, unbounded growth
const cache = {};

function getCached(key) {
  return cache[key];
}

function setCached(key, value) {
  cache[key] = value;
}
```

**Problems:**
- No expiration (memory leak)
- No size limits
- Not thread‑safe in clustered environments
- Lost on process restart

### Production‑Ready In‑Memory Cache with `node-cache`

```javascript
const NodeCache = require('node-cache');

// TTL in seconds, periodic cleanup
const cache = new NodeCache({ 
  stdTTL: 300,           // 5 minutes default TTL
  checkperiod: 60,       // Cleanup every 60 seconds
  useClones: false,      // For better performance (be careful with mutations)
  maxKeys: 10000         // Prevent unbounded growth
});

// Store with custom TTL
cache.set('user:123', userData, 600); // 10 minutes

// Get with automatic TTL refresh (optional)
const user = cache.get('user:123');

// Get multiple keys
const users = cache.mget(['user:123', 'user:456']);

// Delete
cache.del('user:123');

// Flush all
cache.flushAll();

// Get stats
console.log(cache.getStats());
// { hits: 1542, misses: 23, keys: 89, ksize: 2048, vsize: 15240 }
```

### Advanced In‑Memory Cache with TTL and LRU Eviction

```javascript
const LRU = require('lru-cache');

const cache = new LRU({

cache.delete('key');
```

### Tiered Caching: Memory + Disk

For larger caches that survive restarts, combine memory cache with disk persistence:

```javascript
const flatCache = require('flat-cache');
const NodeCache = require('node-cache');

class TieredCache {
  constructor(options = {}) {
    this.memoryCache = new NodeCache({ stdTTL: options.memoryTTL || 300 });
    this.diskCache = flatCache.load(options.cacheId || 'app-cache', options.cacheDir);
    this.persistInterval = options.persistInterval || 60000;
    
    // Load disk cache into memory on startup
    this.loadFromDisk();
    
    // Periodic persistence
    setInterval(() => this.persistToDisk(), this.persistInterval);
  }
  
  loadFromDisk() {
    const all = this.diskCache.all();
    const now = Date.now();
    for (const [key, entry] of Object.entries(all)) {
      if (entry.expires && entry.expires > now) {
        this.memoryCache.set(key, entry.value, (entry.expires - now) / 1000);
      }
    }
  }
  
  persistToDisk() {
    const keys = this.memoryCache.keys();
    for (const key of keys) {
      const value = this.memoryCache.get(key);
      const ttl = this.memoryCache.getTtl(key);
      if (value !== undefined) {
        this.diskCache.setKey(key, {
          value,
          expires: ttl ? Date.now() + (ttl * 1000) : null
        });
      }
    }
    this.diskCache.save(true);
  }
  
  get(key) {
    return this.memoryCache.get(key);
  }
  
  set(key, value, ttl) {
    return this.memoryCache.set(key, value, ttl);
  }
}
```

```javascript
async function getProduct(productId) {

async function getProductWithBackgroundRefresh(productId) {
  const cacheKey = `product:${productId}`;
  
  const entry = cache.get(cacheKey, { allowStale: true });

    // Return stale data immediately
    if (entry.isStale) {
      // Background refresh
      refreshProduct(productId).catch(console.error);
    }
    return entry.value;
  }
  
  // Cache miss — synchronous refresh
  const product = await refreshProduct(productId);
  return product;
}
```

### Caching Patterns with Redis

**1. Cache‑Aside (Lazy Loading)**
```javascript
async function getUser(userId) {
  const cacheKey = `user:${userId}`;
  
  // Try cache first
  const cached = await redis.get(cacheKey);
  if (cached) {
    return JSON.parse(cached);
  }
  
  // Cache miss — query database
  const user = await db.users.findOne({ id: userId });
  
  // Store with TTL (5 minutes)
  if (user) {
    await redis.setex(cacheKey, 300, JSON.stringify(user));
  }
  
  return user;
}
```

**2. Write‑Through Cache**
```javascript
async function updateUser(userId, data) {
  const cacheKey = `user:${userId}`;
  
  // Update database first
  await db.users.update({ id: userId }, data);
  
  // Then update cache (synchronous)
  const updated = await db.users.findOne({ id: userId });
  await redis.setex(cacheKey, 300, JSON.stringify(updated));
  
  return updated;
}
```

**3. Write‑Behind (Write‑Back) Cache**
```javascript
class WriteBehindCache {
  constructor(redis, db, options = {}) {
    this.redis = redis;
    this.db = db;
    this.writeQueue = new Map();
    this.flushInterval = options.flushInterval || 5000;
    this.batchSize = options.batchSize || 100;
    
    setInterval(() => this.flush(), this.flushInterval);
  }
  
  async set(key, value, ttl = 300) {
    // Write to cache immediately
    await this.redis.setex(key, ttl, JSON.stringify(value));
    
    // Queue for async DB write
    this.writeQueue.set(key, { value, timestamp: Date.now() });
    
    if (this.writeQueue.size >= this.batchSize) {
      this.flush();
    }
  }
  
  async flush() {
    if (this.writeQueue.size === 0) return;
    
    const batch = Array.from(this.writeQueue.entries());
    this.writeQueue.clear();
    
    // Batch write to database
    const operations = batch.map(([key, { value }]) => {
      const [prefix, id] = key.split(':');
      return this.db[prefix].upsert({ id, ...value });
    });
    
    await Promise.all(operations);
    console.log(`Flushed ${batch.length} writes to database`);
  }
}
```

## 3. Memoization

Memoization is a **function‑level caching** technique that stores the results of expensive function calls and returns the cached result when the same inputs occur again.

### Basic Memoization Implementation

```javascript
function memoize(fn) {
  const cache = new Map();
  
  return function(...args) {
    const key = JSON.stringify(args);
    
    if (cache.has(key)) {
      console.log('Cache hit');
      return cache.get(key);
    }
    
    console.log('Cache miss — computing');
    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

// Usage
const expensiveCalculation = memoize((n) => {
  console.log(`Computing for ${n}...`);
  let sum = 0;
  for (let i = 0; i < n * 1000000; i++) sum += i;
  return sum;
});

expensiveCalculation(5); // Computes
expensiveCalculation(5); // Returns cached
```

### Memoization with TTL and LRU

```javascript
class Memoizer {
  constructor(options = {}) {
    this.cache = new Map();
    this.ttl = options.ttl || 60000;
    this.maxSize = options.maxSize || 1000;
    this.accessOrder = [];
  }
  
  memoize(fn) {
    return (...args) => {
      const key = JSON.stringify(args);
      const entry = this.cache.get(key);
      const now = Date.now();
      
      // Cache hit and not expired
      if (entry && entry.expires > now) {
        this.updateAccessOrder(key);
        return entry.value;
      }
      
      // Compute
      const result = fn(...args);
      this.set(key, result);
      return result;
    };
  }
  
  set(key, value) {
    // Evict if at capacity
    if (this.cache.size >= this.maxSize) {
      const oldest = this.accessOrder.shift();
      this.cache.delete(oldest);
    }
    
    this.cache.set(key, {
      value,
      expires: Date.now() + this.ttl
    });
    this.updateAccessOrder(key);
  }
  
  updateAccessOrder(key) {
    const index = this.accessOrder.indexOf(key);
    if (index > -1) this.accessOrder.splice(index, 1);
    this.accessOrder.push(key);
  }
  
  clear() {
    this.cache.clear();
    this.accessOrder = [];
  }
}
```

### Async Memoization with Promise Deduplication

```javascript
function memoizeAsync(fn, options = {}) {
  const cache = new Map();
  const pending = new Map(); // Deduplicate in‑flight requests
  const ttl = options.ttl || 60000;
  
  return async function(...args) {
    const key = JSON.stringify(args);
    
    // Check cache
    const entry = cache.get(key);
    if (entry && entry.expires > Date.now()) {
      return entry.value;
    }
    
    // Deduplicate concurrent calls
    if (pending.has(key)) {
      console.log('Awaiting pending request');
      return pending.get(key);
    }
    
    // Execute and store promise for deduplication
    const promise = fn.apply(this, args);
    pending.set(key, promise);
    
    try {
      const result = await promise;
      cache.set(key, {
        value: result,
        expires: Date.now() + ttl
      });
      return result;
    } finally {
      pending.delete(key);
    }
  };
}

// Usage
const fetchUser = memoizeAsync(async (userId) => {
  console.log(`Fetching user ${userId} from API...`);
  const response = await fetch(`https://api.example.com/users/${userId}`);
  return response.json();
}, { ttl: 300000 });

// Multiple concurrent calls only hit API once
await Promise.all([
  fetchUser(123),
  fetchUser(123),
  fetchUser(123)
]); // Only one "Fetching user..." log
```

### Distributed Memoization with Redis

```javascript
class DistributedMemoizer {
  constructor(redis, options = {}) {
    this.redis = redis;
    this.ttl = options.ttl || 300;
    this.prefix = options.prefix || 'memo:';
  }
  
  memoize(fn, keyPrefix = '') {
    return async (...args) => {
      const key = `${this.prefix}${keyPrefix}:${JSON.stringify(args)}`;
      
      // Check Redis cache
      const cached = await this.redis.get(key);
      if (cached) {
        return JSON.parse(cached);
      }
      
      // Compute and store
      const result = await fn(...args);
      await this.redis.setex(key, this.ttl, JSON.stringify(result));
      return result;
    };
  }
  
  async invalidate(pattern) {
    const keys = await this.redis.keys(`${this.prefix}${pattern}`);
    if (keys.length) {
      await this.redis.del(...keys);
    }
  }
}
```

## 4. Database Read Replicas

Read replicas are **read‑only copies** of your primary database that handle read queries, offloading the primary for write operations.

### Architecture Overview

```
                    ┌─────────────────┐
                    │   Application   │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              │              │              │
              ▼              ▼              ▼
       ┌────────────┐ ┌────────────┐ ┌────────────┐
       │   Write    │ │   Read     │ │   Read     │
       │   Primary  │ │  Replica 1 │ │  Replica 2 │
       └────────────┘ └────────────┘ └────────────┘
              │              ▲              ▲
              └──────────────┴──────────────┘
                     Asynchronous Replication
```

### PostgreSQL Read Replica Implementation

```javascript
const { Pool } = require('pg');

class DatabaseRouter {
  constructor() {
    // Write pool (primary)
    this.writePool = new Pool({
      host: process.env.DB_PRIMARY_HOST,
      port: 5432,
      database: process.env.DB_NAME,
      user: process.env.DB_USER,
      password: process.env.DB_PASSWORD,
      max: 20
    });
    
    // Read pools (replicas)
    this.readPools = [
      new Pool({
        host: process.env.DB_REPLICA1_HOST,
        port: 5432,
        database: process.env.DB_NAME,
        user: process.env.DB_READONLY_USER,
        password: process.env.DB_READONLY_PASSWORD,
        max: 10
      }),
      new Pool({
        host: process.env.DB_REPLICA2_HOST,
        port: 5432,
        database: process.env.DB_NAME,
        user: process.env.DB_READONLY_USER,
        password: process.env.DB_READONLY_PASSWORD,
        max: 10
      })
    ];
    
    this.currentReplicaIndex = 0;
  }
  
  // Round‑robin load balancing for reads
  getReadPool() {
    const pool = this.readPools[this.currentReplicaIndex];
    this.currentReplicaIndex = (this.currentReplicaIndex + 1) % this.readPools.length;
    return pool;
  }
  
  async query(sql, params, options = {}) {
    const isWrite = options.write || /^\s*(INSERT|UPDATE|DELETE|CREATE|ALTER|DROP|TRUNCATE)/i.test(sql);
    
    if (isWrite) {
      return this.writePool.query(sql, params);
    } else {
      return this.getReadPool().query(sql, params);
    }
  }
  
  async transaction(callback) {
    const client = await this.writePool.connect();
    try {
      await client.query('BEGIN');
      const result = await callback(client);
      await client.query('COMMIT');
      return result;
    } catch (err) {
      await client.query('ROLLBACK');
      throw err;
    } finally {
      client.release();
    }
  }
}

// Usage
const db = new DatabaseRouter();

// Read query — routed to replica
const users = await db.query('SELECT * FROM users WHERE active = true');

// Write query — routed to primary
await db.query('UPDATE users SET last_login = NOW() WHERE id = $1', [userId], { write: true });

// Transaction — always on primary
await db.transaction(async (client) => {
  await client.query('INSERT INTO orders ...');
  await client.query('UPDATE inventory ...');
});
```

### Handling Replication Lag

Replicas are **eventually consistent** — there's a delay between write and when the replica sees the update.

```javascript
class ReplicationAwareRouter extends DatabaseRouter {
  async query(sql, params, options = {}) {
    const isWrite = options.write || /^\s*(INSERT|UPDATE|DELETE)/i.test(sql);
    
    // Force read from primary when consistency is critical
    if (options.consistent === true) {
      return this.writePool.query(sql, params);
    }
    
    // Use write‑through cache to mitigate replication lag
    if (!isWrite && options.cacheKey) {
      const cached = await this.cache.get(options.cacheKey);
      if (cached) return cached;
    }
    
    const result = await super.query(sql, params, options);
    
    // After write, cache the expected result for immediate read consistency
    if (isWrite && options.cacheAfterWrite) {
      await this.cache.set(options.cacheAfterWrite.key, options.cacheAfterWrite.value, 5);
    }
    
    return result;
  }
}
```

## 5. Database Sharding

Sharding splits a large database into smaller, independent pieces (**shards**) distributed across multiple servers.

### Sharding Strategies

| Strategy | How It Works | Best For |
|----------|--------------|----------|
| **Range‑Based** | Partition by key ranges (e.g., user 1‑1000 on shard1) | Time‑series data, sequential IDs |
| **Hash‑Based** | Hash the shard key, distribute evenly | Uniform distribution, avoiding hotspots |
| **Directory‑Based** | Lookup table maps keys to shards | Complex relationships, flexibility |
| **Geo‑Based** | Partition by geographic region | Multi‑region applications |

### Application‑Level Sharding with Consistent Hashing

```javascript
const crypto = require('node:crypto');

class ConsistentHash {
  constructor(nodes = [], virtualNodes = 150) {
    this.nodes = [];
    this.ring = new Map(); // hash -> node
    this.sortedHashes = [];
    this.virtualNodes = virtualNodes;
    
    nodes.forEach(node => this.addNode(node));
  }
  
  hash(key) {
    return crypto.createHash('md5').update(key).digest('hex').substring(0, 8);
  }
  
  addNode(node) {
    this.nodes.push(node);
    for (let i = 0; i < this.virtualNodes; i++) {
      const virtualKey = `${node}:${i}`;
      const hash = this.hash(virtualKey);
      this.ring.set(hash, node);
      this.sortedHashes.push(hash);
    }
    this.sortedHashes.sort();
  }
  
  removeNode(node) {
    this.nodes = this.nodes.filter(n => n !== node);
    for (let i = 0; i < this.virtualNodes; i++) {
      const virtualKey = `${node}:${i}`;
      const hash = this.hash(virtualKey);
      this.ring.delete(hash);
      this.sortedHashes = this.sortedHashes.filter(h => h !== hash);
    }
  }
  
  getNode(key) {
    if (this.sortedHashes.length === 0) return null;
    
    const keyHash = this.hash(key);
    
    // Binary search for the first hash >= keyHash
    let left = 0, right = this.sortedHashes.length - 1;
    while (left <= right) {
      const mid = Math.floor((left + right) / 2);
      if (this.sortedHashes[mid] < keyHash) {
        left = mid + 1;
      } else {
        right = mid - 1;
      }
    }
    
    const index = left % this.sortedHashes.length;
    const hash = this.sortedHashes[index];
    return this.ring.get(hash);
  }
}
```

### Sharded Database Connection Pool

```javascript
class ShardedDatabase {
  constructor(shardConfigs) {
    this.shards = new Map();
    this.consistentHash = new ConsistentHash(shardConfigs.map(c => c.name));
    
    shardConfigs.forEach(config => {
      this.shards.set(config.name, new Pool({
        host: config.host,
        database: config.database,
        user: config.user,
        password: config.password
      }));
    });
  }
  
  getShard(shardKey) {
    const shardName = this.consistentHash.getNode(shardKey);
    return {
      name: shardName,
      pool: this.shards.get(shardName)
    };
  }
  
  async query(shardKey, sql, params) {
    const { name, pool } = this.getShard(shardKey);
    console.log(`Routing to shard: ${name}`);
    return pool.query(sql, params);
  }
  
  // Cross‑shard query (expensive — avoid in production)
  async queryAllShards(sql, params) {
    const results = [];
    for (const [name, pool] of this.shards) {
      const result = await pool.query(sql, params);
      results.push({ shard: name, rows: result.rows });
    }
    return results;
  }
}

// Usage
const db = new ShardedDatabase([
```

### Resharding Strategy (Zero‑Downtime Migration)

```javascript
class MigratingShardedDatabase extends ShardedDatabase {
  constructor(shardConfigs, migrationConfig = {}) {
    super(shardConfigs);
    this.migrationMode = migrationConfig.enabled || false;
    this.newConsistentHash = migrationConfig.newHashRing;
  }
  
  async query(shardKey, sql, params) {
    if (this.migrationMode) {
      const oldShard = this.consistentHash.getNode(shardKey);
      const newShard = this.newConsistentHash.getNode(shardKey);
      
      if (oldShard !== newShard) {
        // Dual‑write during migration
        const [oldResult, newResult] = await Promise.allSettled([
          this.shards.get(oldShard).query(sql, params),
          this.shards.get(newShard).query(sql, params)
        ]);
        
        // Read from old shard until cutover
        return oldResult.status === 'fulfilled' ? oldResult.value : newResult.value;
      }
    }
    
    return super.query(shardKey, sql, params);
  }
}
```

## 6. Write‑Through vs Write‑Back Caching

### Write‑Through Cache

Data is written to **both cache and database** synchronously. The write is considered complete only when both succeed.

```
    Write Request
         │
         ▼
    ┌─────────┐
    │  Cache  │ ◀── Write to cache first
    └────┬────┘
         │
         ▼
    ┌─────────┐
    │Database │ ◀── Then write to database
    └─────────┘
         │
         ▼
    Response to client
```

**Advantages:**
- Strong consistency — cache always reflects database
- No data loss on cache failure
- Simple to implement

**Disadvantages:**
- Higher latency (two writes)
- Cache doesn't reduce write load on database

```javascript
class WriteThroughCache {
  constructor(cache, db) {
    this.cache = cache;
    this.db = db;
  }
  
  async set(key, value, ttl = 300) {
    // Write to database first (or cache first — both synchronous)
    await this.db.set(key, value);
    
    // Then update cache
    await this.cache.setex(key, ttl, JSON.stringify(value));
    
    return value;
  }
  
  async get(key) {
    // Read from cache
    const cached = await this.cache.get(key);
    if (cached) {
      return JSON.parse(cached);
    }
    
    // Cache miss — read from DB and populate cache
    const value = await this.db.get(key);
    if (value) {
      await this.cache.setex(key, 300, JSON.stringify(value));
    }
    return value;
  }
}
```

### Write‑Back (Write‑Behind) Cache

Data is written **only to cache** initially. The database is updated **asynchronously** later.

```
    Write Request
         │
         ▼
    ┌─────────┐
    │  Cache  │ ◀── Write to cache only (fast)
    └────┬────┘
         │
         ▼
    Response to client (immediate)
         
    ... Later (async) ...
         
    ┌─────────┐
    │  Cache  │
    └────┬────┘
         │
         ▼
    ┌─────────┐
    │Database │ ◀── Batch write to database
    └─────────┘
```

**Advantages:**
- Very low write latency
- Reduced database write load (batching)
- Can absorb write spikes

**Disadvantages:**
- Risk of data loss if cache fails before flush
- Eventual consistency (stale reads possible)
- More complex implementation

```javascript
class WriteBackCache {
  constructor(cache, db, options = {}) {
    this.cache = cache;
    this.db = db;
    this.writeQueue = new Map();
    this.flushInterval = options.flushInterval || 5000;
    this.batchSize = options.batchSize || 100;
    this.maxQueueSize = options.maxQueueSize || 10000;
    
    // Periodic flush
    this.flushTimer = setInterval(() => this.flush(), this.flushInterval);
  }
  
  async set(key, value, ttl = 300) {
    // Write to cache immediately
    await this.cache.setex(key, ttl, JSON.stringify(value));
    
    // Queue for async database write
    this.writeQueue.set(key, {
      value,
      timestamp: Date.now(),
      attempts: 0
    });
    
    // Backpressure: if queue is too large, force synchronous flush
    if (this.writeQueue.size >= this.maxQueueSize) {
      console.warn('Write queue full — forcing synchronous flush');
      await this.flush();
    } else if (this.writeQueue.size >= this.batchSize) {
      // Trigger async flush (don't await)
      this.flush().catch(console.error);
    }
    
    return value;
  }
  
  async flush() {
    if (this.writeQueue.size === 0) return;
    
    const batch = Array.from(this.writeQueue.entries());
    this.writeQueue.clear();
    
    console.log(`Flushing ${batch.length} writes to database`);
    
    const operations = batch.map(async ([key, { value }]) => {
      try {
        await this.db.set(key, value);
      } catch (err) {
        console.error(`Failed to write ${key}:`, err);
        // Re‑queue with backoff
        const entry = this.writeQueue.get(key) || { value, timestamp: Date.now(), attempts: 0 };
        if (entry.attempts < 3) {
          entry.attempts++;
          this.writeQueue.set(key, entry);
        } else {
          console.error(`Dropping ${key} after 3 failed attempts`);
        }
      }
    });
    
    await Promise.all(operations);
  }
  
  async get(key) {
    // Always read from cache
    const cached = await this.cache.get(key);
    if (cached) {
      return JSON.parse(cached);
    }
    
    // Cache miss — read from database
    const value = await this.db.get(key);
    if (value) {
      await this.cache.setex(key, 300, JSON.stringify(value));
    }
    return value;
  }
  
  async shutdown() {
    clearInterval(this.flushTimer);
    await this.flush(); // Final flush
  }
}
```

### Write‑Around Cache

Data is written **directly to database**, bypassing the cache. The cache is only populated on read misses.

```
    Write Request
         │
         ▼
    ┌─────────┐
    │Database │ ◀── Write directly (cache unchanged)
    └─────────┘
         │
         ▼
    Response to client
```

**Best for:** Write‑heavy workloads where written data is rarely read immediately.

```javascript
class WriteAroundCache {
  async set(key, value) {
    // Write only to database
    await this.db.set(key, value);
    // Optionally invalidate cache
    await this.cache.del(key);
  }
  
  async get(key) {
    // Read from cache
    let value = await this.cache.get(key);
    if (value) return JSON.parse(value);
    
    // Cache miss — read from DB and cache it
    value = await this.db.get(key);
    if (value) {
      await this.cache.setex(key, 300, JSON.stringify(value));
    }
    return value;
  }
}
```

### Comparison Table

| Pattern | Write Latency | Consistency | Data Loss Risk | Database Write Load |
|---------|--------------|-------------|----------------|---------------------|
| **Write‑Through** | High | Strong | None | High |
| **Write‑Back** | Low | Eventual | Medium (cache failure) | Low (batched) |
| **Write‑Around** | Medium | Eventual | None | High (but cache not updated) |
| **Cache‑Aside** | Medium | Eventual | None | Medium |

### Real‑World: Hybrid Strategy with TTL‑Based Promotion

```javascript
class HybridCache {
  constructor(options = {}) {
    this.writeBack = new WriteBackCache(options.redis, options.db, {
      flushInterval: 5000,
      batchSize: 100
    });
    this.writeThrough = new WriteThroughCache(options.redis, options.db);
  }
  
  async set(key, value, options = {}) {
    // Critical data uses write‑through
    if (options.critical) {
      return this.writeThrough.set(key, value, options.ttl);
    }
    
    // Non‑critical data uses write‑back for speed
    return this.writeBack.set(key, value, options.ttl);
  }
  
  async get(key) {
    // Same read path for both
    return this.writeThrough.get(key);
  }
}
```

## Summary Table: Caching Strategies Decision Matrix

| Scenario | Recommended Strategy |
|----------|---------------------|
| Read‑heavy, data changes rarely | Cache‑Aside with long TTL |
| Write‑heavy, reads are infrequent | Write‑Around |
| Need immediate read‑after‑write consistency | Write‑Through |
| High write throughput, can tolerate slight data loss | Write‑Back |
| Session data, user‑specific | In‑Memory + Redis (distributed) |
| API rate limiting | Redis (atomic counters) |
| Leaderboards, real‑time rankings | Redis Sorted Sets |
| Large dataset, single DB overloaded | Sharding + Read Replicas |
| Expensive function calls | Memoization |

**Key Takeaways:**
1. **Cache at multiple layers** — memory (fastest), Redis (distributed), CDN (edge)
2. **Always set TTLs** — prevent unbounded memory growth
3. **Use read replicas** to scale read‑heavy workloads
4. **Shard by a natural key** (user ID, tenant ID) for horizontal scaling
5. **Choose write strategy based on consistency requirements**
6. **Monitor cache hit ratio** — low ratio means ineffective caching
7. **Implement circuit breakers** for cache failures to prevent DB overload
