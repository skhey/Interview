# Advanced Database Integration – Node.js Guide

Database integration in Node.js goes far beyond simple CRUD operations. Production applications demand **connection pooling** for scalability, **transactions** for data integrity, **isolation levels** for concurrency control, and **advanced query patterns** for complex data processing. This guide covers the essential patterns for working with relational databases (PostgreSQL, MySQL) and MongoDB in high‑performance Node.js applications.

---

## 1. Connection Pooling

Creating a new database connection for every request is **extremely inefficient**. Connection pooling maintains a reusable pool of connections, dramatically improving throughput and reducing latency.

### Why Connection Pooling Matters

| Without Pooling | With Pooling |
|-----------------|--------------|
| TCP handshake per query | Reuse existing connections |
| High latency overhead | Sub‑millisecond connection acquisition |
| Database connection limit exhaustion | Controlled concurrency |
| Resource churn | Stable resource usage |

### PostgreSQL Connection Pool with `pg`

```javascript
const { Pool } = require('pg');

const pool = new Pool({
  host: process.env.DB_HOST,
  port: process.env.DB_PORT,
  database: process.env.DB_NAME,
  user: process.env.DB_USER,
  password: process.env.DB_PASSWORD,
  max: 20,                         // Maximum connections in pool
  idleTimeoutMillis: 30000,        // Close idle connections after 30s
  connectionTimeoutMillis: 2000,   // Fail if can't connect within 2s
  maxUses: 7500,                   // Rotate connections after 7500 uses
  allowExitOnIdle: true,
  ssl: process.env.NODE_ENV === 'production' ? { rejectUnauthorized: false } : false
});

// Monitor pool events
pool.on('connect', (client) => {
  console.log('New client connected to database');
});

pool.on('acquire', (client) => {
  console.log('Client acquired from pool');
});

pool.on('remove', (client) => {
  console.log('Client removed from pool');
});

pool.on('error', (err, client) => {
  console.error('Unexpected error on idle client', err);
});

// Usage
async function getUser(id) {
  const client = await pool.connect();
  try {
    const result = await client.query('SELECT * FROM users WHERE id = $1', [id]);
    return result.rows[0];
  } finally {
    client.release(); // Always release back to pool
  }
}

// Simple query (acquires and releases automatically)
async function getUsers() {
  const result = await pool.query('SELECT * FROM users');
  return result.rows;
}
```

```javascript
const pool = mysql.createPool({

  return rows[0];
}

// Transaction with connection from pool
async function transferMoney(fromId, toId, amount) {
  const connection = await pool.getConnection();
  try {
    await connection.beginTransaction();
    await connection.execute(
      'UPDATE accounts SET balance = balance - ? WHERE id = ?',
      [amount, fromId]
    );
    await connection.execute(
      'UPDATE accounts SET balance = balance + ? WHERE id = ?',
      [amount, toId]
    );
    await connection.commit();
  } catch (err) {
    await connection.rollback();
    throw err;
  } finally {
    connection.release();
  }
}
```

### Pool Sizing Guidelines

| Metric | Recommendation |
|--------|----------------|
| **Max Connections** | Start with `(CPU cores * 2) + effective_spindle_count`; monitor and adjust |
| **Idle Timeout** | 30–60 seconds (balance between quick reuse and resource freeing) |
| **Connection Lifetime** | Rotate after 1–2 hours or N queries to prevent memory leaks |
| **Queue Timeout** | Fail fast (2–5 seconds) rather than indefinite waiting |

## 2. Transactions

Transactions ensure that a series of database operations either **all succeed** or **all fail**, maintaining data integrity.

### ACID Properties

| Property | Meaning | Database Guarantee |
|----------|---------|-------------------|
| **Atomicity** | All or nothing | Transaction rollback on error |
| **Consistency** | Valid state before and after | Constraints, triggers |
| **Isolation** | Concurrent transactions don't interfere | Isolation levels |
| **Durability** | Committed data survives crashes | WAL, fsync |

### Basic Transaction with PostgreSQL

```javascript
async function transferFunds(fromAccount, toAccount, amount) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Check sufficient balance
    const balanceRes = await client.query(
      'SELECT balance FROM accounts WHERE id = $1 FOR UPDATE',
      [fromAccount]
    );
    if (balanceRes.rows[0].balance < amount) {
      throw new Error('Insufficient funds');
    }

    // Perform transfer
    await client.query(
      'UPDATE accounts SET balance = balance - $1 WHERE id = $2',
      [amount, fromAccount]
    );
    await client.query(
      'UPDATE accounts SET balance = balance + $1 WHERE id = $2',
      [amount, toAccount]
    );

    // Record transaction
    await client.query(
      'INSERT INTO transfers (from_account, to_account, amount, created_at) VALUES ($1, $2, $3, NOW())',
      [fromAccount, toAccount, amount]
    );

    await client.query('COMMIT');
    return { success: true };
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### Transaction with Savepoints

Savepoints allow partial rollbacks within a transaction.

```javascript
async function complexOrderProcessing(orderData) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Create order
    const orderRes = await client.query(
      'INSERT INTO orders (user_id, total) VALUES ($1, $2) RETURNING id',
      [orderData.userId, orderData.total]
    );
    const orderId = orderRes.rows[0].id;

    // Set savepoint before potentially failing operations
    await client.query('SAVEPOINT order_items');

    try {
      for (const item of orderData.items) {
        await client.query(
          'INSERT INTO order_items (order_id, product_id, quantity) VALUES ($1, $2, $3)',
          [orderId, item.productId, item.quantity]
        );
        await client.query(
          'UPDATE inventory SET stock = stock - $1 WHERE product_id = $2',
          [item.quantity, item.productId]
        );
      }
    } catch (err) {
      // Rollback only order items, keep the order header
      await client.query('ROLLBACK TO SAVEPOINT order_items');
      await client.query(
        'UPDATE orders SET status = $1 WHERE id = $2',
        ['failed', orderId]
      );
    }

    // Release savepoint
    await client.query('RELEASE SAVEPOINT order_items');

    await client.query('COMMIT');
    return { orderId };
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### Distributed Transactions (Two‑Phase Commit)

For operations spanning multiple databases or services, use the Saga pattern (covered in Microservices guide) or a transaction coordinator.

```javascript
// Simplified 2PC coordinator (production use a dedicated system like Atomikos or Temporal)
class TwoPhaseCommitCoordinator {
  constructor(participants) {
    this.participants = participants;
  }

  async execute(transactionId, work) {
    // Phase 1: Prepare
    const prepared = [];
    for (const participant of this.participants) {
      const canCommit = await participant.prepare(transactionId, work[participant.name]);
      if (!canCommit) {
        await this.rollback(transactionId, prepared);
        throw new Error(`Participant ${participant.name} failed to prepare`);
      }
      prepared.push(participant);
    }

    // Phase 2: Commit
    for (const participant of prepared) {
      await participant.commit(transactionId);
    }
  }

  async rollback(transactionId, prepared) {
    for (const participant of prepared) {
      await participant.rollback(transactionId);
    }
  }
}
```

### MongoDB Transactions (Replica Set Required)

```javascript
const { MongoClient } = require('mongodb');

async function transferFundsMongo(fromAccount, toAccount, amount) {
  const client = await MongoClient.connect(process.env.MONGO_URL);
  const session = client.startSession();

  try {
    session.startTransaction();

    const accounts = client.db().collection('accounts');

    // Check balance
    const from = await accounts.findOne(
      { _id: fromAccount },
      { session }
    );
    if (from.balance < amount) {
      throw new Error('Insufficient funds');
    }

    // Update accounts
    await accounts.updateOne(
      { _id: fromAccount },
      { $inc: { balance: -amount } },
      { session }
    );
    await accounts.updateOne(
      { _id: toAccount },
      { $inc: { balance: amount } },
      { session }
    );

    // Record transfer
    await client.db().collection('transfers').insertOne({
      fromAccount,
      toAccount,
      amount,
      createdAt: new Date()
    }, { session });

    await session.commitTransaction();
  } catch (err) {
    await session.abortTransaction();
    throw err;
  } finally {
    session.endSession();
    await client.close();
  }
}
```

## 3. Read Isolation Levels

Isolation levels control **how much a transaction is isolated** from concurrent transactions. Understanding them is crucial for balancing performance and consistency.

### Standard SQL Isolation Levels

| Level | Dirty Read | Non‑Repeatable Read | Phantom Read | Performance |
|-------|------------|---------------------|--------------|-------------|
| **Read Uncommitted** | Possible | Possible | Possible | Fastest |
| **Read Committed** (PostgreSQL default) | No | Possible | Possible | Good |
| **Repeatable Read** | No | No | Possible (PG: No) | Slower |
| **Serializable** | No | No | No | Slowest |

### Setting Isolation Level in PostgreSQL

```javascript
async function runWithIsolation(isolationLevel, work) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');
    await client.query(`SET TRANSACTION ISOLATION LEVEL ${isolationLevel}`);

    const result = await work(client);

    await client.query('COMMIT');
    return result;
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}

// Read Committed (default) — sees only committed data
await runWithIsolation('READ COMMITTED', async (client) => {
  const user = await client.query('SELECT * FROM users WHERE id = $1', [id]);
  // May see different data if another transaction commits during this one
});

// Repeatable Read — sees snapshot from transaction start
await runWithIsolation('REPEATABLE READ', async (client) => {
  const first = await client.query('SELECT * FROM users WHERE id = $1', [id]);
  // ... other operations ...
  const second = await client.query('SELECT * FROM users WHERE id = $1', [id]);
  // first and second are guaranteed identical
});

// Serializable — highest isolation
await runWithIsolation('SERIALIZABLE', async (client) => {
  // PostgreSQL may throw serialization failure; retry needed
  const result = await client.query('SELECT COUNT(*) FROM users');
});
```

### Handling Serialization Failures

```javascript
async function runSerializableWithRetry(work, maxRetries = 3) {
  for (let attempt = 1; attempt <= maxRetries; attempt++) {
    try {
      return await runWithIsolation('SERIALIZABLE', work);
    } catch (err) {
      if (err.code === '40001' || err.code === '40P01') {
        // Serialization failure — retry
        console.log(`Serialization failure, retry ${attempt}/${maxRetries}`);
        if (attempt === maxRetries) throw err;
        // Exponential backoff
        await new Promise(r => setTimeout(r, Math.pow(2, attempt) * 100));
      } else {
        throw err;
      }
    }
  }
}
```

### Read Isolation in MongoDB

MongoDB uses **Read Concern** and **Write Concern** instead of traditional isolation levels.

```javascript
// Read concern "majority" — only see data replicated to majority
const user = await collection.findOne(
  { _id: userId },
  { readConcern: { level: 'majority' } }
);

// Read concern "snapshot" (within transaction) — consistent snapshot
const session = client.startSession();
session.startTransaction({
  readConcern: { level: 'snapshot' },
  writeConcern: { w: 'majority' }
});
```

### Choosing the Right Isolation Level

| Scenario | Recommended Level |
|----------|-------------------|
| High‑volume reads, no writes during read | Read Committed |
| Report generation requiring consistent snapshot | Repeatable Read |
| Financial transactions requiring absolute correctness | Serializable |
| Inventory management (prevent overselling) | Repeatable Read + SELECT FOR UPDATE |

## 4. WAL Logs (Write‑Ahead Logging)

WAL is a **durability mechanism** used by PostgreSQL, MySQL (InnoDB), and many other databases. Before any change is written to data files, it's **appended to the WAL**. This ensures that after a crash, the database can replay the WAL to recover to a consistent state.

### How WAL Works

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           Write‑Ahead Logging Process                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                              │
│  1. Transaction begins                                                       │
│                                                                              │
│  2. Changes written to WAL buffer (in memory)                                │
│                                                                              │
│  3. COMMIT forces WAL buffer to disk (fsync)                                 │
│                                                                              │
│  4. Transaction considered durable                                           │
│                                                                              │
│  5. Background writer eventually flushes changes to data files               │
│                                                                              │
│  6. Checkpoint marks point where all data files are up‑to‑date               │
│                                                                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### WAL Configuration in PostgreSQL

```sql
-- View current WAL settings
SHOW wal_level;        -- 'replica' (default) or 'logical'
SHOW max_wal_size;     -- Maximum WAL size before automatic checkpoint
SHOW wal_buffers;      -- Shared memory for WAL (default -1 = auto)
SHOW checkpoint_timeout; -- Max time between checkpoints
SHOW synchronous_commit; -- 'on' (safe), 'off' (fast), 'remote_write', etc.

-- Adjust for performance
ALTER SYSTEM SET wal_buffers = '16MB';
ALTER SYSTEM SET max_wal_size = '4GB';
ALTER SYSTEM SET checkpoint_timeout = '15min';
SELECT pg_reload_conf();
```

### WAL and Replication

WAL is the foundation for PostgreSQL **streaming replication**. Standby servers continuously replay WAL from the primary.


### Monitoring WAL in Node.js

```javascript
async function getWALMetrics() {
  const result = await pool.query(`
    SELECT
      pg_current_wal_lsn() AS current_lsn,
      pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0') AS bytes_written,
      (SELECT COUNT(*) FROM pg_ls_waldir()) AS wal_files,
      pg_size_pretty(
        pg_wal_lsn_diff(pg_current_wal_lsn(), '0/0')
      ) AS wal_size
  `);
  return result.rows[0];
}

// Expose as Prometheus metric
setInterval(async () => {
  const metrics = await getWALMetrics();
  walBytesGauge.set(metrics.bytes_written);
}, 30000);
```

### WAL Implications for Application Developers

- **Synchronous commit** (`synchronous_commit = on`) guarantees durability but adds latency. Consider `remote_write` for replication with lower latency.
- **Large transactions** generate WAL proportional to the size of changes — batch carefully.
- **WAL archiving** enables point‑in‑time recovery (PITR).

## 5. SELECT FOR UPDATE (Row‑Level Locking)

`SELECT ... FOR UPDATE` locks selected rows, preventing concurrent modifications. It's essential for **implementing safe counters, inventory management, and financial operations**.

### How It Works

When you execute `SELECT ... FOR UPDATE` within a transaction, PostgreSQL acquires **row‑level exclusive locks** on the returned rows. Other transactions attempting to `SELECT ... FOR UPDATE` or `UPDATE` those rows will **block** until the lock is released.

### Basic Usage

```javascript
async function reserveInventory(productId, quantity) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Lock the inventory row
    const res = await client.query(
      `SELECT stock FROM inventory 
       WHERE product_id = $1 
       FOR UPDATE`,
      [productId]
    );

    const currentStock = res.rows[0]?.stock || 0;
    if (currentStock < quantity) {
      throw new Error('Insufficient stock');
    }

    // Update safely — no race condition
    await client.query(
      'UPDATE inventory SET stock = stock - $1 WHERE product_id = $2',
      [quantity, productId]
    );

    await client.query('COMMIT');
    return { success: true, remaining: currentStock - quantity };
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### SELECT FOR UPDATE Variants

| Clause | Behavior |
|--------|----------|
| `FOR UPDATE` | Blocks until lock acquired; strong lock |
| `FOR NO KEY UPDATE` | Weaker lock; allows concurrent non‑key updates |
| `FOR SHARE` | Allows reads, blocks exclusive locks |
| `FOR KEY SHARE` | Weakest lock; for foreign key checks |
| `SKIP LOCKED` | Skip already‑locked rows (queue processing) |
| `NOWAIT` | Fail immediately if row locked |

### SKIP LOCKED for Queue Processing

```javascript
// Worker processing jobs without blocking
async function claimNextJob() {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    const res = await client.query(`
      SELECT id, payload 
      FROM jobs 
      WHERE status = 'pending' 
      ORDER BY created_at 
      LIMIT 1 
      FOR UPDATE SKIP LOCKED
    `);

    if (res.rows.length === 0) {
      await client.query('ROLLBACK');
      return null;
    }

    const job = res.rows[0];
    await client.query(
      'UPDATE jobs SET status = $1, started_at = NOW() WHERE id = $2',
      ['processing', job.id]
    );

    await client.query('COMMIT');
    return job;
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

### Avoiding Deadlocks with SELECT FOR UPDATE

```javascript
// Always lock resources in consistent order
async function transfer(fromId, toId, amount) {
  const client = await pool.connect();
  try {
    await client.query('BEGIN');

    // Lock rows in consistent order (smaller ID first)
    const [first, second] = fromId < toId ? [fromId, toId] : [toId, fromId];

    await client.query(
      'SELECT * FROM accounts WHERE id IN ($1, $2) ORDER BY id FOR UPDATE',
      [first, second]
    );

    // Now safe to update in any order
    await client.query('UPDATE accounts SET balance = balance - $1 WHERE id = $2', [amount, fromId]);
    await client.query('UPDATE accounts SET balance = balance + $1 WHERE id = $2', [amount, toId]);

    await client.query('COMMIT');
  } catch (err) {
    await client.query('ROLLBACK');
    throw err;
  } finally {
    client.release();
  }
}
```

## 6. Aggregation Pipelines

Aggregation pipelines transform and summarize data. While MongoDB popularized the concept, SQL databases offer equally powerful aggregation capabilities.

### MongoDB Aggregation Pipeline

```javascript
const pipeline = [
  // Stage 1: Match active users
  { $match: { status: 'active', createdAt: { $gte: new Date('2024-01-01') } } },

  // Stage 2: Lookup orders
  { $lookup: {
    from: 'orders',
    localField: '_id',
    foreignField: 'userId',
    as: 'orders'
  } },

  // Stage 3: Unwind orders
  { $unwind: { path: '$orders', preserveNullAndEmptyArrays: true } },

  // Stage 4: Group by user and calculate metrics
  { $group: {
    _id: '$_id',
    name: { $first: '$name' },
    email: { $first: '$email' },
    totalSpent: { $sum: '$orders.total' },
    orderCount: { $sum: { $cond: [{ $ifNull: ['$orders._id', false] }, 1, 0] } },
    avgOrderValue: { $avg: '$orders.total' },
    lastOrderDate: { $max: '$orders.createdAt' }
  } },

  // Stage 5: Filter high‑value customers
  { $match: { totalSpent: { $gte: 1000 } } },

  // Stage 6: Sort
  { $sort: { totalSpent: -1 } },

  // Stage 7: Add rank
  { $setWindowFields: {
    sortBy: { totalSpent: -1 },
    output: { rank: { $rank: {} } }
  } },

  // Stage 8: Limit
  { $limit: 100 }
];

const highValueUsers = await db.collection('users').aggregate(pipeline).toArray();
```

### PostgreSQL Aggregation with Window Functions

```sql
WITH user_orders AS (
  SELECT
    u.id,
    u.name,
    u.email,
    COALESCE(SUM(o.total), 0) AS total_spent,
    COUNT(o.id) AS order_count,
    AVG(o.total) AS avg_order_value,
    MAX(o.created_at) AS last_order_date
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  WHERE u.status = 'active'
    AND u.created_at >= '2024-01-01'
  GROUP BY u.id, u.name, u.email
),
ranked_users AS (
  SELECT
    *,
    RANK() OVER (ORDER BY total_spent DESC) AS rank
  FROM user_orders
)
SELECT *
FROM ranked_users
WHERE total_spent >= 1000
ORDER BY total_spent DESC
LIMIT 100;
```

### Complex Aggregation with PostgreSQL JSON Functions

```javascript
// Using JSON aggregation for nested data
const result = await pool.query(`
  SELECT
    u.id,
    u.name,
    u.email,
    COALESCE(
      JSON_AGG(
        JSON_BUILD_OBJECT(
          'orderId', o.id,
          'total', o.total,
          'items', (
            SELECT JSON_AGG(
              JSON_BUILD_OBJECT(
                'productId', oi.product_id,
                'quantity', oi.quantity,
                'price', oi.price
              )
            )
            FROM order_items oi
            WHERE oi.order_id = o.id
          )
        )
        ORDER BY o.created_at DESC
      ) FILTER (WHERE o.id IS NOT NULL),
      '[]'::json
    ) AS orders
  FROM users u
  LEFT JOIN orders o ON u.id = o.user_id
  WHERE u.id = $1
  GROUP BY u.id
`, [userId]);

// Returns user with nested orders array
```

### Aggregation Performance Tips

| Database | Tip |
|----------|-----|
| **MongoDB** | Use indexes that cover `$match` and `$sort` stages; `$lookup` benefits from indexes on foreign field |
| **PostgreSQL** | Use `EXPLAIN ANALYZE` to identify bottlenecks; create indexes on `GROUP BY` and `WHERE` columns |
| **General** | Pre‑aggregate data with materialized views for frequently‑run reports |

### Materialized Views for Pre‑computed Aggregates

```sql
-- PostgreSQL materialized view
CREATE MATERIALIZED VIEW user_metrics AS
SELECT
  u.id,
  u.name,
  COUNT(o.id) AS order_count,
  COALESCE(SUM(o.total), 0) AS total_spent
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;

-- Create index for fast queries
CREATE INDEX idx_user_metrics_total_spent ON user_metrics(total_spent);

-- Refresh manually or on schedule
REFRESH MATERIALIZED VIEW CONCURRENTLY user_metrics;
```

```javascript
// Node.js scheduled refresh
const cron = require('node-cron');

cron.schedule('0 */6 * * *', async () => {
  await pool.query('REFRESH MATERIALIZED VIEW CONCURRENTLY user_metrics');
  console.log('Materialized view refreshed');
});
```

| **Aggregation Pipelines** | Complex data transformation | MongoDB aggregation, SQL `GROUP BY` with window functions |

**Key Takeaways:**
1. **Always use connection pooling** — creating connections per request is a scalability anti‑pattern
2. **Wrap related operations in transactions** — ensure data consistency
3. **Choose the right isolation level** — default Read Committed is usually sufficient
4. **Use `SELECT FOR UPDATE`** for safe counters and inventory operations
5. **Leverage aggregation pipelines** — push computation to the database, not the application
6. **Monitor pool metrics and WAL** — observability prevents surprises in production
